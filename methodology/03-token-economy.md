# Token Economy

> **Status:** Operational convention. The whole point of having structure is to read less, not more.
> Last updated: 2026-07-15

A personal workspace accumulates documents. Without rules, every agent that enters the workspace reads everything. That's expensive — and worse, it dilutes attention. The token economy is the set of rules that decide what gets read when.

---

## The problem

Without a token economy, every agent that enters a workspace reads everything by default. That's expensive (because re-reading is wasteful) and worse (because it dilutes attention). A specific instance of this problem, observed in one OperatorOS-style workspace before this convention was adopted, was ~12K tokens per cold-start session — most of that cost was reading the same files over and over to rediscover structure that didn't change.

After applying the token economy in that same instance: ~2-4K tokens for cold-start, ~0.5K for a follow-up session that already knows the workspace.

That's roughly a 70-80% reduction. The methodology (the four reading tiers below) is the lever; the percentages above are illustrative of one engineer's observed outcome and are not guaranteed across all workspaces.

---

## The four reading tiers

| Tier | When read | Approx. tokens |
|---|---|---|
| **Bootstrap** | Every session, every agent, no exceptions. | ~2K |
| **Conditional** | Only when the task references the concept. | ~1-3K each |
| **Discovery** | Only when an unknown concept is encountered. | ~1-5K each |
| **Ignore** | Never. Not by agents, not by humans reading casually. | 0 |

### Bootstrap (always read)

These four files (or their equivalents) form the bootstrap:

1. `IDENTITY.md` — who the workspace owner is, what they do, what they care about.
2. `BOOTSTRAP.md` (or `bootstrap.md`) — the file you're reading conceptually: how to navigate this workspace.
3. `system-graph.md` — the high-level map: which subsystems exist, how they relate.
4. `workspace-index.json` — the machine-readable registry.

Total: ~2K tokens. Read this on every cold-start. Don't skip.

### Conditional (read when relevant)

Most of the workspace. Concrete examples vary by engineer — the pattern is "a subsystem directory at the workspace root is read only when its topic matches the current task":

- Subsystem directory `<your-subsystem-a>/` — read only if the task involves that subsystem's domain.
- Subsystem directory `<your-subsystem-b>/` — read only if the task involves that subsystem's artifacts.
- A `strategic-analyses/` directory at the workspace root — read only when the user explicitly references it or asks for analysis.

The agent should ask: "Is this document relevant to the current task?" If no, skip it. The concrete subsystem names in any given workspace are the engineer's choice — the framework does not specify which subsystems a workspace should have.

### Discovery (read on demand)

When the agent encounters a concept it doesn't recognize (e.g., a project name, a tool, a person), it follows the cross-reference chain. This costs tokens but only when needed.

Example: the task mentions "OperatorOS" and the workspace contains a dedicated `CONTEXT/operatoros/` directory tracking OperatorOS-related notes. The agent discovers it and reads that, then decides whether to dive deeper.

### Ignore (never read)

- `.git/`, `node_modules/`, `dist/`, `build/` — generated content.
- Files in `ARCHIVE/` unless explicitly requested.
- Test files unless the task is about testing.
- Generated docs (e.g., `docs/api/` built from source).

The bootstrap file declares the ignore list. The agent respects it without question.

## The bootstrap file

A workspace generated by OperatorOS includes a `bootstrap.md` at the root. This file is the agent's entry point. It has four sections:

```markdown
# Bootstrap

## 1. Always read first (bootstrap)
- IDENTITY.md (owner context)
- system-graph.md (subsystem map)
- workspace-index.json (machine registry)
- BOOTSTRAP.md (this file or its equivalent)

## 2. Read if relevant (conditional)
- <subsystem>/README.md — read if task touches <subsystem>
- ...

## 3. Discover on demand
- Run search_files for concepts not in the registry.
- Cross-references chain. Cost is bounded by task scope.

## 4. Never read (ignore)
- .git/, node_modules/, dist/, build/
- ARCHIVE/ (unless explicitly requested)
- <list of always-ignore patterns>
```

The bootstrap file is generated by `operatoros init` and updated by `operatoros add` / `operatoros validate`. The agent reads it first.

---

## The interview

When the bootstrap protocol doesn't give the agent enough context, it runs the onboarding interview (see `05-onboarding-interview.md`). This costs ~0.5K tokens and happens once per agent-workspace pair.

After the interview, the agent writes the answers to `IDENTITY.md` (or an override file), so future cold-starts don't need to re-ask.

## How to apply this to your own workspace

1. **Run `operatoros init`.** This generates a default bootstrap file.
2. **Edit `bootstrap.md`** to reflect your actual workspace structure.
3. **Move large documents to conditional tier** if they're rarely needed.
4. **Add ignore patterns** for generated content, archives, and noise.
5. **Test it.** Have a fresh agent cold-start in your workspace. Time it. Count tokens. If cold-start > 5K tokens, your bootstrap is too heavy.

---

## Common pitfalls

- **The everything-bootstrap.** Putting every important file in the bootstrap tier. Cold-start balloons to 10K+ tokens. Move rarely-read docs to conditional.
- **The nothing-bootstrap.** Bootstrap has only one file. Agent can't navigate; re-discovers everything. Cold-start is fast but subsequent reads are slow.
- **The ignored-important-file.** A critical document is in the ignore list. Agent doesn't read it. Fix by reviewing the ignore list quarterly.
- **The stale-bootstrap.** Bootstrap file hasn't been updated since the workspace was first created. New subsystems are ignored. Run `operatoros validate` to detect drift.

---

## Token economy as a design constraint

When you add a document to your workspace, ask:

1. **Who needs to read this?** (Bootstrap / conditional / discovery / ignore)
2. **How often?** (Every session / rarely / once)
3. **What if it's never read?** (Is it still useful as a reference?)

If the answers are "conditional, rarely, yes", put it in conditional tier. If they're "discovery, once, yes", put it in discovery. If they're "ignore, never, no", delete it.

Most workspaces have too many "discovery" docs that should be "ignore". Be ruthless.
