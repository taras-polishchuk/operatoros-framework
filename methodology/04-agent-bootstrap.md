# Agent Bootstrap Protocol

> **Status:** Operational protocol. This is what an AI agent does when it enters your workspace for the first time.

The bootstrap protocol is the contract between a workspace and an AI agent. The workspace says: "Here is what you should read, in what order, and when to ask me questions." The agent says: "I will respect this contract."

A workspace that doesn't have a bootstrap protocol forces every agent to rediscover structure from scratch. That's expensive, slow, and error-prone.

---

## The protocol, step by step

When an agent enters a workspace generated by OperatorOS, it does the following:

### Step 0 — Locate the bootstrap file

Look for `bootstrap.md` at the workspace root. If absent, look for `BOOTSTRAP.md`, `.bootstrap.md`, or `AGENTS.md`. If none exist, treat the workspace as unstructured and apply minimum-effort scanning.

### Step 1 — Read the bootstrap file

The bootstrap file declares:
- The four files to read first (bootstrap tier)
- The conditional files (read when relevant)
- The discovery rules (search on demand)
- The ignore list

Read the entire bootstrap file. It should be ~1-2K tokens.

### Step 2 — Read the bootstrap tier

Read the four files declared in section 1 of the bootstrap file. This is ~2K tokens total. After this step, the agent knows:

- Who owns this workspace
- What subsystems exist
- How to find any document
- What to ignore

### Step 3 — Decide whether to run the onboarding interview

Check `IDENTITY.md` (or equivalent) for the `onboarding_complete: true` flag.

- If present: skip to Step 4.
- If absent: run the onboarding interview (see `05-onboarding-interview.md`). Write the answers to `IDENTITY.md`. Cost: ~0.5K tokens.

### Step 4 — Wait for the task

The agent is now ready. It should not read additional files until the user provides a task.

### Step 5 — On task receipt

When the user gives a task:

1. Read the bootstrap file again (cheap; ~1-2K tokens; guarantees freshness).
2. Identify which conditional files are relevant.
3. Read those. Don't read unrelated conditional files.
4. If a concept is unknown, run a discovery search.
5. Do the task.
6. Update mission artifacts if the task is non-trivial (see `methodology/02-doc-lifecycle.md` §"Mission artifacts" and `methodology/06-decisions-adr.md`).

### Step 6 — On session end

If the agent made any state changes (created files, modified configs), it should:

1. Run `operatoros validate` (if a CLI is available) to verify schema conformance.
2. Update `progress.md` in the relevant mission directory.
3. Commit the changes (if the workspace is a git repo and the agent is authorized).

## What the bootstrap file must contain

A valid bootstrap file has these sections, in this order:

```markdown
# Bootstrap — <workspace name>

## 1. Always read first
- <path 1>: <what it tells the agent>
- <path 2>: <what it tells the agent>
- <path 3>: <what it tells the agent>
- <path 4>: <what it tells the agent>

## 2. Read when relevant
- <path>: <trigger condition>
- <path>: <trigger condition>

## 3. Discover on demand
- <search pattern>: <what to look for>

## 4. Never read
- <pattern 1>: <reason>
- <pattern 2>: <reason>

## 5. Onboarding
- If `IDENTITY.md` lacks `onboarding_complete: true`, run the interview
- Interview script: `methodology/05-onboarding-interview.md`
```

---

## What the bootstrap file must NOT contain

- **The whole system-graph inline.** Reference it. Don't repeat it.
- **Mission-specific details.** Those go in mission artifacts, not bootstrap.
- **Code snippets or examples.** Reference docs that have them.
- **User-private data.** No personal notes, no credentials, no financial data.

The bootstrap file is the front door. Keep it clean.

---

## Generated vs hand-written

OperatorOS generates a default bootstrap file on `init`. The user customizes it.

**The default is not optimal.** It's a starting point. Customize it for your actual workspace.

**Don't auto-regenerate the bootstrap file.** It should only change when the workspace structure changes, and that should be a deliberate decision.

## Multi-agent workspaces

When multiple agents work in the same workspace, the bootstrap file is shared. Each agent has its own `IDENTITY.md` override file:

```
IDENTITY.md           # shared owner context
IDENTITY.<agent>.md   # agent-specific context (Claude, Aider, Hermes, ...)
```

Each agent reads the shared `IDENTITY.md` and its own override. The overrides contain:

- Agent name and version
- Last session timestamp
- Persistent context (e.g., "this agent works on the `<subsystem>` directory")
- Onboarding answers (if onboarding is per-agent, not shared)

## Failure modes

**The over-helpful agent.** Reads everything in bootstrap tier + most conditional + runs full discovery. Result: cold-start 15K tokens. Fix: agent must respect the bootstrap file's tier structure.

**The under-helpful agent.** Reads only the bootstrap file. Doesn't read conditional files when relevant. Result: agent asks the user for information that's already in the workspace. Fix: agent must check the conditional list before asking.

**The stale bootstrap.** Agent reads a bootstrap file from 2024 in a workspace that's changed since. Result: agent ignores the new subsystems. Fix: workspace should run `operatoros validate` regularly to flag stale bootstraps.

**The bootstrap-without-workspace.** Agent is given only the bootstrap file but no workspace access. Result: agent hallucinates. Fix: agent must verify it can read the declared paths before relying on them.

---

## How to test the protocol

1. Start a fresh agent in your workspace.
2. Give it a simple task: "Summarize the workspace structure."
3. Time the response.
4. Count tokens consumed.
5. Verify the agent's summary matches reality.

A well-bootstrapped agent should complete this task in < 5K tokens and produce an accurate summary.

If it takes > 10K tokens or the summary is wrong, your bootstrap file needs work.
