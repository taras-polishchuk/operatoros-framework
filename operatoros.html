The problem

You are one operator. You run a workspace that is supposed to be yours — files, scripts, notes, tools, AI agents, scheduled jobs, integrations. In practice it has fragmented into:

  • A ~/projects/ tree that has drifted since 2023
  • Three different note systems, none of which the others can read
  • AI agents that each invented their own conventions
  • A bash history that is the actual source of truth for 40% of your workflows
  • Secrets scattered across .env, gpg, and that one Notion page

None of this is a single system. None of it is replaceable. None of it composes.

The hypothesis

If DevOps was the discipline of running production software at the team level, then the missing discipline for the next decade is the operating layer for the single-operator level — one human, their tools, their agents, their data.

OperatorOS is that discipline, captured as a framework.

Six principles

01 · Single Authority

Every concept has one canonical owner file. No dual sources. If two files disagree, one is wrong.

02 · Everything Replaceable

Modules, presets, tools, even Core itself — all replaceable. Framework first, implementation last.

03 · Typed Substrate

Every artifact validated against a JSON Schema at the boundary. Invalid state cannot exist.

04 · AI-native

Core ships ≥1 of: agent-loop primitive, structured-output validation, prompt-template versioning, LLM-provider abstraction.

05 · Composable Modules

A module is one composable extension. It declares a contract (module.yaml), runs in isolation, exposes one CLI surface.

06 · Profile-portable

Your whole workspace exports to a single bundle. Move to a new machine in 10 minutes. Publish, fork, or archive at will.

Quickstart (v0.3.0-alpha)

# Option A — install the single-file binary (recommended)
curl -fsSL https://raw.githubusercontent.com/taras-polishchuk/operatoros-framework/main/scripts/install.sh | sh
# operatoros is now on your PATH

# Option B — clone + build from source
git clone https://github.com/taras-polishchuk/operatoros-framework.git
cd operatoros-framework/core
npm install && npm run build

# 1. initialize a personal workspace
operatoros init --personal
cd personal-workspace

# 2. validate against the JSON-Schema
operatoros validate operatoros.yaml

# 3. apply the preset (installs all modules in presets/personal/preset.yaml)
operatoros apply
# → installs journal module (the only example)

# 4. run a module command
operatoros run journal add "shipped v0.3.0-alpha — apply + run + binary"

# 5. export the workspace (deny-list excludes vault/, .env, *.sqlite)
operatoros export --bundle tar.gz
# → personal-workspace-2026-07-02T21-30.tar.gz (secrets excluded)

# 6. version
operatoros version

v0.3.0-alpha ships 6 commands: init / validate / add / apply / run / export. 18 tests passing. ~746 KB single-file binary via ncc. v1.0 will add a built-in module registry and a one-line install for the binary.

What it is NOT for

OperatorOS is not for teams — Backstage is. It is not for SaaS deployment — Kubernetes is. It is not for hosting websites — Vercel is. It is not a graph database, a static site generator, a chat framework, or an agent runtime — those are modules you install on top of OperatorOS.

OperatorOS is for one operator: the human (and their AI agents) who want their workspace to be a system, not a folder.
